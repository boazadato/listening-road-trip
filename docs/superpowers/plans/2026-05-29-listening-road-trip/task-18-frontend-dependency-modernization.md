## Task 18: Frontend Dependency Modernization (Tier B: full current)

**Status:** ⬜ deferred. Reverses the frontend-major freeze from **Revision Note 3** — see **Revision Note 8** in the index for the decision and the spike evidence.

**Prerequisites:** Tasks 1–17 complete (the app ships and is deployed). `make dev` must run for the E2E verification in Step 3.

This task bumps the frontend's frozen majors to current. It was **validated empirically before being written**: a throwaway git-worktree spike applied every bump below and ran `pnpm install` + `tsc --noEmit` + `vitest run` + `vite build` with **zero source changes** — all green (peers clean, `tsc` exit 0, 3/3 tests, build OK). So this is expected to be a near-pure `package.json` change. The risk that remains is **runtime**, not compile-time (see Step 3).

**Files:**
- `frontend/package.json` — version bumps only (no source edits expected).
- `pnpm-lock.yaml` — regenerated by `pnpm install`.
- Source edits only if the live app surfaces a runtime regression in Step 3.

---

- [ ] **Step 1: Apply the version bumps**

Edit `frontend/package.json`:

```jsonc
// dependencies
"react": "^19.0.0",
"react-dom": "^19.0.0",
"react-router-dom": "^7.0.0",
"zustand": "^5.0.0",

// devDependencies
"@types/react": "^19.0.0",
"@types/react-dom": "^19.0.0",
"@vitejs/plugin-react": "^5.0.0",
"vite": "^6.0.0",
"vitest": "^4.1.0"
```

Then from the repo root:

```bash
corepack pnpm install
```

Expected (verified in the spike): no peer-dependency errors. `@cloudflare/vitest-pool-workers` in the worker keeps the worker on its own vitest 4 — the two packages resolve independently under pnpm workspaces, so the monorepo simply unifies on a single vitest major.

- [ ] **Step 2: Static gates (both packages)**

```bash
cd frontend && npx tsc --noEmit && npx vitest run && npx vite build
cd worker   && npx tsc --noEmit && pnpm test
```

Expected (verified in the spike, frontend side): `tsc` exit 0, 3/3 tests pass, build succeeds, and the Vite "CJS Node API deprecated" warning is **gone**. The production bundle grows ≈30% gzipped (≈67 KB → ≈87 KB) — expected, from React 19 + RR7.

- [ ] **Step 3: Runtime verification — re-run the Task 16 Playwright E2E golden path**

**This is the gate that lets the marker flip to ✅, and it is not optional.** Tier B swaps the React + router *runtime*; the static gates in Step 2 only prove there are no type/build breaks. The 3 unit tests cover `CountdownTimer`/`tripStore` and do **not** exercise the flows most exposed to React 19 / RR7 behavioral changes (StrictMode double-invoke, ref/effect timing, router internals):

1. Start `make dev`.
2. Drive the [Task 16](task-16-build-playwright-e2e.md) golden path with the `mcp__playwright__*` tools: create-trip form + flavour pickers → join in a second context → `/ws` upgrade + `state_sync` → **reconnect toast** appears on Worker stop and clears on restart → Leaderboard empty state → Analysis locked.
3. If Spotify + Claude creds are in `worker/.dev.vars`, also run the live AI-DJ flow (OAuth → song push → rate → reveal).

Fix any runtime regression with the smallest source change and re-run. If React 19 StrictMode surfaces a double-effect in `useWebSocket` (double WS connect), that is the most likely hotspot — confirm the cleanup path closes the socket idempotently.
